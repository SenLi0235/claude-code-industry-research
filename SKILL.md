# 行业研究 v5.1.0 — Industry Research

一份合格的行研报告 = **系统的分析框架 + 可溯源的数据 + 透明的计算过程 + 诚实的不确定性披露**。

本 skill 只回答一个问题：**这个行业能不能进？** 不回答"投哪家"。如需标的筛选可在完成后用 `/screen-ideas`。

## 0. Pre-flight Gate（四步串行，禁止跳过）

详见 `references/pre-flight-gate.md`。摘要与分工：

| 步骤 | 做什么 | 谁做 |
|---|---|---|
| **0.1** | AskUserQuestion 一次问完 5 项：行业+口径 / 目的 / 资料路径 / 输出路径 / 时间预算 | 主 agent |
| **0.1b** | 技术分析模式判定（**默认硬科技**，完整技术分析 + S2 API；仅消费/周期/简单制造 → 取消硬科技） | 主 agent |
| **0.2** | 三通道验证：WebSearch（必选，系统内置）/ Wind（推荐，万得金融数据，走 Skill 工具；每日免费1000积分）/ S2（推荐，Semantic Scholar学术引擎，免费注册API Key） | 主 agent |
| **0.3** | 创建主报告骨架 + 研究笔记 + TaskCreate | 主 agent |
| **0.4** | 本地资料 → MD → 索引（≤3份且总≤5万字：主agent直接Read；否则派Explore子agent并发生成索引） | 主 agent / Explore |

每步完成前不得进入下一步。完成后向用户口头确认四步 OK。

## 1. 工作流 — 3 次审批窗口

```
Window A （轻量审批）
  ├─ §2.0 行业定义 + §2.1 生命周期  → 同一个 subagent 连写
  └─ ★ 用户确认：定义边界和阶段判断是否正确

Window B （关键审批）  ← 最核心改进
  ├─ §2.2 产业链与价值链  → 先跑（低风险，不阻塞）
  ├─ §2.3 竞争格局  → 先交 6-10 行提纲，用户批准后写正文
  ├─ §2.4 市场规模  → 先交 6-10 行提纲，用户批准后写正文
  │   （§2.3 和 §2.4 提纲获批后可并发写正文）
  └─ ★ 用户确认：提纲（竞品分组/定位可视化 + 市场情景假设/测算方法），然后看正文

Window C （批量审批）
  ├─ §2.5 PEST + §2.6 景气度  → 并发派两个 subagent
  ├─ 主 agent 撰写核心结论（≤800字，3-5条）
  ├─ 主 agent 合并附录 A（矛盾+[UNSOURCED]）/ B（待补充）/ C（参考资料）
  ├─ 用户审阅通过后 → AskUserQuestion 询问是否生成 HTML 报告
  └─ ★ 用户最终审阅全文
```

### 为什么这样改

对比旧版（v4.3.0）每板块独立派工+同一套验证、7 次汇报、串行执行：

| 旧版 | v5.0.0 | 理由 |
|---|---|---|
| 每节同级别抽检 | 高风险做提纲审批，低风险抽检即可 | §2.4 算错增速 vs §2.5 漏一条政策，影响完全不同 |
| 发现无源数据 → 退回阻塞 | 标 `[UNSOURCED]` 继续走 | 不确定性是输出属性，不是流程阻塞 |
| 强制串行（编号连号） | 预分配编号段，允许 Window B/C 内并发 | 编号段消除了串行的唯一理由 |
| 7 次用户确认 | 3 次窗口 | 减少疲劳，合并同类审批 |
| 闸门问完不再交互 | 中程 AskUserQuestion 确认关键决策 | 竞品分组逻辑、情景假设——让用户参与判断 |

## 2. `[UNSOURCED]` 标记机制

**找不到可靠来源的数字**：不要阻塞流程，在正文中标注 `[UNSOURCED]`，继续往下走。

- 单个数值无源 → 标 `[UNSOURCED]`，记入笔记 §A（注明"待用户确认或替换"）
- 板块内 `[UNSOURCED]` 占比 >30% → 退回重写（说明检索深度不够）
- 附录 A 统一列出所有 `[UNSOURCED]` 项，用户最后一次性处理

禁止的行为：
- ❌ 编造数字（比 `[UNSOURCED]` 更差）
- ❌ 用"约/大约/据估算"模糊掩饰
- ❌ 因为一个 `[UNSOURCED]` 而卡住整个板块

## 3. 预分配编号段（消除串行依赖）

| 板块 | 编号段 | 可并发 |
|---|---|---|
| §2.0 + §2.1 | [1-10] | — |
| §2.2 | [11-35] | —（§2.3/§2.4 的前置） |
| §2.3 | [36-60] | 与 §2.4 并发（提纲获批后） |
| §2.4 | [61-90] | 与 §2.3 并发（提纲获批后） |
| §2.5 | [91-110] | 与 §2.6 并发 |
| §2.6 | [111-130] | 与 §2.5 并发 |

每个 subagent 只在自己的编号段内追加，不再依赖追加顺序保证连号。

## 4. 差异化审批标准

| 板块 | 风险 | 审批方式 |
|---|---|---|
| §2.0 行业定义 | 低 | Window A 合并审，Read 抽检前 20 行 |
| §2.1 生命周期 | 低 | Window A 合并审，Read 抽检前 20 行 |
| §2.2 产业链 | 中 | 先跑不阻塞；标准抽检（前 40+后 10 行） |
| §2.3 竞争格局 | **高** | **先交提纲 → 用户批准 → 写正文 → 标准抽检** |
| §2.4 市场规模 | **高** | **先交提纲 → 用户批准 → 写正文 → 标准抽检** |
| §2.5 PEST | 低 | 并发派工，抽检前 20 行 |
| §2.6 景气度 | 低 | 并发派工，抽检前 20 行 |
| 核心结论 | — | 主 agent 撰写，Window C 统一审 |

## 5. Mid-Process 交互点

Window B 中，收到提纲后用 AskUserQuestion 确认以下决策：

**§2.3 竞争格局提纲确认**：
- 竞品分组逻辑（按商业模式 / 客户群 / 姿态 / 出身）？
- 定位可视化选型（2×2矩阵 / 雷达图 / 层级图 / 价值链图）？
- 竞品列表是否遗漏/多余？

**§2.4 市场规模提纲确认**：
- 测算方法（需求导向 / 供给导向 / 引用现成报告）？
- 三情景核心假设是否合理？
- 是否有特定数据源要求？

## 6. Subagent 派工模板

派工前：主 agent 必须 Read 对应的 `skills/<name>/SKILL.md`，将其【板块模板】段完整注入 subagent prompt。

```
你是行业研究 skill 的板块研究员，负责写【§2.X 板块名】。

【上下文】
- 行业：<名称+口径> | 研究目的：<...> | 时间预算：<1h/1d/深度>
- 生命周期阶段：<...> | 技术模式：<硬科技/基础>
- 代表企业（锚点，不展开对标）：<3-5家跨环节企业>
- 资料索引：<绝对路径，无则填"无">
- 主报告：<绝对路径> | 研究笔记：<绝对路径>
- **你的编号段**：[X-Y] | 起始编号 = X

【检索通道】
- WebSearch（必选，系统内置）| Wind（推荐，走 Skill 工具调用 wind-mcp-skill，金融/宏观/行业结构化数据；每日免费1000积分）| Semantic Scholar（推荐，S2 API Key: $S2_API_KEY，免费注册，1 RPS）
- 路由：国内政策/行业数据/中文内容→WebSearch主力；海外/全球对标/英文报告→WebSearch主力+S2辅助；金融财务/宏观指标/行业统计→Wind主力（无Wind时WebSearch+公开财报兜底）；技术/论文→S2主力+WebSearch交叉验证
- 四轮检索：广搜验证 → 矛盾深挖 → 建模交叉 → 案例补全

【板块模板】
<主 agent 从 skills/<name>/SKILL.md 复制完整模板填入>

【工作流要求】
1. 资料索引处理：先 Read 索引 → 按相关性挑 1-3 文件 → offset+limit ≤200 行
2. 每轮检索后 Bash >> 追记关键词到笔记 §D
3. WebFetch 抓取有价值 URL 时只抽数字/结论，≤300字
4. 每条可引用数据 Bash >> 笔记 §C（格式：[N] 机构《文件名》年份, URL, 数值=X）
5. 找不到可靠来源 → 正文标 [UNSOURCED] + 追加笔记 §A（不要阻塞）
6. 发现数据矛盾 → 追加笔记 §A；信息缺口 → 追加笔记 §B
7. 用 Edit 写入主报告对应章节（占位符已被创建）

【返回格式】仅返回 ≤200 字摘要：
- 完成（是/否）
- 新增来源 N 条（编号 [a]-[b]）/ [UNSOURCED] N 条 / 矛盾 a 条 / 缺口 b 条
- 1-2 句话核心发现
- 阻塞问题（有/无，描述）
不要回传正文内容、不要回传搜索结果原文。
```

对于 §2.3 和 §2.4（高风险板块），subagent 返回摘要前**先返回提纲**：
```
【提纲】（6-10行）
- 本板块要回答的核心问题
- 拟采用的数据来源/测算方法
- 关键假设
- 输出结构（节标题 + 一句话内容备注）
```
主 agent 将提纲呈现给用户，批准后再让 subagent 继续写正文。

## 9. 禁止行为清单

- ❌ 跳过 §0 四道闸门
- ❌ 主 agent 亲自做检索/WebFetch/写板块正文（只有 §0 + Phase C 合并 + 核心结论除外）
- ❌ 把 WebFetch 当搜索引擎用（仅用于读已知 URL）
- ❌ 整报告一次性写完才落盘
- ❌ 使用未在笔记 §C 登记的数字
- ❌ 发现无源数字时不标 `[UNSOURCED]` 直接写进去
- ❌ 在行业研究报告中展开公司级分析（商业模式UE/对标/估值/投资建议）
- ❌ 用"约/大约/据估计"模糊词代替定量数据
- ❌ 照抄资料整段文字（必须复述+分析；引用超 3 句用引用块并注页码）

## 10. HTML 报告输出

> **依赖声明**：HTML 生成依赖 `frontend-design` 技能（Claude Code 内置），请确保已启用。

Window C 用户审阅通过后，主 agent 用 AskUserQuestion 询问是否生成 HTML 报告。

### 生成流程

1. AskUserQuestion："报告已定稿，是否生成 HTML 网页版？（依赖 frontend-design 技能）"
2. 用户确认 → 派 subagent，注入 `frontend-design` skill + `references/html-report-design.md` 完整视觉规范
3. Subagent 派工 prompt：

```
你是行业研究 skill 的 HTML 报告生成器。使用 frontend-design 技能，按以下规范生成单个 HTML 文件：

【视觉规范】必须完整 Read references/html-report-design.md，严格遵守色彩/字体/布局/组件规范。
【输入】Read 主报告 MD 文件（<绝对路径>）。
【输出】Write 单个 HTML 文件到 <主报告同目录>/<行业名>-行业研究报告.html。

技术要求：
- 单文件、零外部依赖、CSS/JS 全部内联
- 必须适配 @media print（A4），深色/浅色模式切换
- 左侧固定 TOC 导航，IntersectionObserver 高亮当前位置
- 数据表格 sticky header、隔行底纹、数字右对齐 tabular-nums
- KPI 卡片、结论框、[UNSOURCED] 标记、引用跳转 严格按规范渲染
- 所有数字和结论必须来自报告原文，不添加任何新内容
```

4. Subagent 落盘后，主 agent 告知用户文件路径。

### 禁止行为

- ❌ 擅自改视觉规范（色彩/字体/布局已固定）
- ❌ 添加报告中不存在的数据或结论
- ❌ 引入外部 CDN 依赖（字体用系统栈，不需要 Google Fonts）
- ❌ 在 MD 报告定稿前生成 HTML
